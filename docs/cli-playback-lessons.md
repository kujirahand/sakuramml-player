# CLI 再生まわりの学び

CLI 版の再生ノイズ対応を通して、今回の実装で分かったことを整理する。

## 何が起きていたか

- ブラウザ版ではノイズが出ないのに、CLI 版だけプチプチとクリックノイズが入った
- WAV 書き出し結果は正常だった
- つまり、PCM 生成ロジックそのものよりも、CLI のリアルタイム再生経路に原因があると考えられた

## 原因の整理

### 1. `rodio::Source` の扱いは雑に実装すると破綻しやすい

`Source::current_frame_len()` は単なるバッファ残量ではなく、`rodio` 側がフォーマット変化の境界として扱う情報に近い。

ここを短いチャンク単位で返すと、変換器や内部処理が細かく区切られ、クリックノイズの原因になりうる。

### 2. CLI の出力デバイス sample rate は固定ではない

ブラウザ版は `AudioContext.sampleRate` を使うため、実デバイスに近い条件で動く。

一方 CLI 版を 44100 Hz 固定にすると、macOS 側の既定出力が 48000 Hz だった場合に再サンプルが発生する。このときチャンク境界の扱いが悪いとノイズが目立ちやすい。

### 3. 「オーディオスレッドで都度レンダリング」は安全ではない

逐次 `next()` の中でレンダリングを進める形は単純に見えるが、オーディオ再生の要求タイミングと重い処理が直結する。

今回のように Rust 側でシンセ処理を行う場合、供給が少しでも不安定だとクリックノイズや途切れにつながりやすい。

## 切り分けで有効だったこと

### WAV 書き出しとの比較

同じ `Player` と `Synth` を使って WAV が正常に書き出せるなら、音源や MIDI 解析本体よりも、リアルタイム再生経路を優先して疑える。

### 実機 sample rate の確認

macOS の出力デバイス設定を確認し、既定が 48000 Hz であることを把握できたのは大きかった。

「ブラウザでは平気なのに CLI だけ壊れる」場合、デバイス側の sample rate 差は強い手掛かりになる。

### 最も単純な再生経路を一度作る

全曲を先にレンダリングして `SamplesBuffer` で鳴らす経路を一度作ると、リアルタイム供給由来かどうかを強く切り分けられる。

今回も `--render-all` 相当の挙動でノイズが消えたため、問題がストリーミング経路にあると判断しやすくなった。

## 最終的に採った設計

### 既定はストリーム再生

- 起動待ちを短くしたい
- すべての曲を最初にレンダリングすると体感待ち時間が長い
- メモリ使用量も曲長に比例して増える

このため、通常はストリーム再生を基本にする。

### ストリーム再生は別スレッド先読み方式にする

- 再生スレッド側は、できるだけ軽くサンプルを取り出すだけにする
- PCM 生成はワーカースレッド側で先読みする
- チャンクをある程度バッファしておく

これにより、再生タイミングと音源レンダリングを分離できる。

### `--render-all` を逃げ道として残す

- ノイズ切り分けをしたいとき
- ストリーム再生で環境依存の問題が疑われるとき
- 起動待ちよりも安全性を優先したいとき

この用途では、全曲先レンダリング再生は有効な保険になる。

## 今後の運用指針

- 既定 CLI 再生はストリーム方式を使う
- 問題の再現確認や保守的な再生が必要なら `--render-all` を使う
- CLI とブラウザの再生差を疑うときは、まず sample rate とバッファ供給方式を見る
- WAV が正常なら、音源本体よりもリアルタイム出力経路を先に疑う

## 実装面の教訓

- オーディオ処理では「正しく鳴る」だけでなく「どのスレッドで何をしているか」が重要
- 再生開始速度、ノイズ耐性、メモリ消費はトレードオフになる
- デバッグ用に「全曲先レンダリング」の経路を持っておくと切り分けが速い
- ブラウザ版と CLI 版で同じ `Player` を共有していても、出力先の違いだけで不具合の出方は変わる
