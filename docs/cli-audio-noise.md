# CLI 再生ノイズ調査メモ

CLI 版でだけ再生中にプチプチとノイズが出る件を調べた結果、原因は PCM 生成本体ではなく、`src/main.rs` の `rodio::Source` ラッパーにありました。

## 症状

- ブラウザ版ではノイズが出ない
- WAV 書き出しでもノイズの原因になるような破綻は見えない
- CLI のリアルタイム再生でのみプチプチが乗る

## 原因

### 1. `current_frame_len()` の使い方が `rodio` の想定とずれていた

`rodio` の `Source::current_frame_len()` は、単なる「今のバッファ残量」ではなく、次にフォーマットが変わりうるまでの残りサンプル数を返す前提です。

今回の旧実装では、`PlayerSource` が持つ 0.1 秒ぶんのチャンク残量をここで返していました。そのため `rodio` 側の変換器からは、短い区切りで別フレームが連続する入力に見えてしまいます。

### 2. CLI 側だけ 44100 Hz 固定だった

ブラウザ版は `AudioContext.sampleRate` を Rust 側へ渡していますが、CLI 版は 44100 Hz 固定でした。

macOS の既定出力デバイスが 48000 Hz の環境では、`rodio` 側でサンプルレート変換が入ります。このとき `current_frame_len()` が細かいチャンク境界を示していると、変換器の再初期化境界でクリックノイズが出やすくなります。

## 修正方針

- `current_frame_len()` は `None` を返し、単一フォーマットの連続ストリームとして `rodio` に渡す
- 出力デバイスの既定サンプルレートを取得し、その値で `Player` を初期化する
- 既定の CLI 再生は、別スレッドで PCM を先読みするストリーム再生にする
- 必要な場合だけ `--render-all` で全曲 PCM を作ってから `SamplesBuffer` で流す

## 切り分け結果

- `Player::render_next()` のテストは通る
- 同じ曲を WAV へ書き出すと正常に完了する
- したがって、共有の PCM 生成ロジックよりも CLI 専用の出力経路を優先して直すのが妥当だった

## 実装後の方針

ブラウザ版はメモリ節約のためにチャンク先読みを続ける。CLI 版は通常時は別スレッドで先読みするストリーム再生を使い、起動待ちを減らす。

`--render-all` を付けた場合だけ、曲全体を先にレンダリングしてから再生する。これにより、必要に応じて最も保守的な再生経路も選べるようにする。
