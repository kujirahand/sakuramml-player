# MIDI メタイベントの扱いとテキスト解析における知見

MIDIファイルの再生や解析（特にRustとJavaScriptを連携させたアーキテクチャ）において、MIDIのメタイベントに関する情報を抽出する際に得られた重要な知見をまとめます。

## テキストイベント (0x01 〜 0x07) について

MIDIフォーマットにおいて、文字列データを持つメタイベント群（`MetaType: 0x01 〜 0x07`）は以下のような種類に分かれています。

- `0x01` : Text (任意のテキストイベント)
- `0x02` : Copyright (著作権情報)
- `0x03` : Sequence/Track Name (トラック名や曲名)
- `0x04` : Instrument Name (楽器名)
- `0x05` : Lyric (歌詞)
- `0x06` : Marker (マーカー、Aメロ・Bメロなどのセクション指定によく使われる)
- `0x07` : Cue Point (キューポイント)

MMLコンパイラ（例：sakuramml）や、一般的なMIDI作成ソフトによって、これらのイベントにテキストが埋め込まれます。

## 実装上の注意点と落とし穴

バイナリとして抽出した生バイト配列から文字列へ変換する際、以下の点に注意する必要があります。

### 1. Null文字（`\0`）による終端

一部のシーケンサーやコンパイラ（MMLを含む）が生成するMIDIファイルでは、テキストイベントの終端を C言語のような**Null文字列 (`\0`) でパディングまたは終端**しているケースがあります。
Rustの文字列処理やJavaScriptの文字列処理において、このNull文字が混入してしまうと以下の問題を引き起こします。

- Visual / DOM 表示時に空文字列として扱われてしまう。
- Web側にJSONとして渡したときにブラウザのUIに描画されない（コンソールでは見えない制御文字として出力され、処理が壊れる）。
- 抽出結果が空と見間違えられやすい（`console.log`で空っぽに見えることがある）。

**解決策:**  
抽出したテキストデータに対して `trim_matches(char::from(0))` のように、末尾や文字列中の Null 文字を取り除くサニタイズ処理を行う必要があります。

```rust
// Rust側でのサニタイズ例
let mut text = match std::str::from_utf8(text_bytes) {
    Ok(s) => s.to_string(),
    Err(_) => {
        let (cow, _, _) = encoding_rs::SHIFT_JIS.decode(text_bytes);
        cow.into_owned()
    }
};
// 必須処理：Null文字を取り除く
text = text.trim_matches(char::from(0)).to_string();
```

### 2. 文字エンコーディング (UTF-8 vs Shift_JIS)

標準的なMIDIファイル (SMF) は、古い日本の楽曲データやフリーで配布されているファイルの大半が **Shift_JIS** で記述されています。しかし、モダンなMMLソフトはUTF-8で出力するものもあります。

- 最初に `from_utf8` でパースを試みる。
- 失敗した場合に備えて、`encoding_rs` などのクレートを用いて `Shift_JIS` へのフォールバックを実装しておくアプローチが極めて有効かつ堅牢です。

### 3. 表示目的に合わせたフロントエンド側への振り分け

テキストイベントを無差別にすべて描画するとUIがカオスになります。全抽出（`0x01..=0x07`）をバックエンド側で行い、フロントエンド（JS）で役割に応じて振り分ける設計が望ましいです。

- **`0x06` (Marker) や `0x01` (Text)**: ピアノロール上にタイムラインと一緒にマッピングする。
- **`0x05` (Lyric)**: リアルタイムで再生バーに合わせて歌詞専用ラベルに表示する。
- **`0x03` (Track Name)**: デバッグやログ用として保持し、画面のタイトルバーに使う。

このように、責務をRust(全抽出とパース)・JS(分類と描画)にしっかり分けることで拡張性の高いプレイヤーを実装することができます。

## フロントエンドでの歌詞表示の扱い

歌詞を専用ラベルに都度表示する方式は単純ですが、シーク時に「今どこまで表示済みか」という状態が DOM に分散しやすくなります。今回のプレイヤーのようにコンパイルログ欄を併用する場合は、以下のように扱うと破綻しにくくなります。

- コンパイルログの基底文字列を別変数で保持する。
- Lyric の表示は「追記」ではなく、現在時刻までの歌詞一覧を毎回再構築して textarea に反映する。
- 再生中だけでなく、停止やシーク時にも同じ再構築関数を呼ぶ。

この形にしておくと、再生を巻き戻した時もログ欄の内容を正しく戻せます。単純な append だけで進めると、シークバック時に古い歌詞が残り続けるので注意が必要です。
